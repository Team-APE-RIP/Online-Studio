# Team-APE-RIP-Online-Studio 设计思路

## 1. 项目定位

**Team-APE-RIP-Online-Studio（Team APE:RIP Online Studio）** 是一款运行在 **Linux + Docker** 环境中的在线工作室平台，数据库使用 **MySQL**，部署目标适配 **宝塔 Linux 面板**，应用服务运行在 **14010** 端口，并通过宝塔可视化管理 MySQL（3306 端口）与容器编排配置。

平台的核心定位不是普通在线文档系统，而是：

> **以 HOI4 Modding 协作为核心场景、兼顾多组织 SaaS 扩展能力的在线工作室平台。**

它既要具备类似腾讯文档的多人在线实时协作体验，又要支持类似 GitHub Organizations 的团队、组织、权限和项目协作能力，并针对 HOI4 模组开发过程中的核心对象进行建模与编辑。

---

## 2. 核心目标

平台需要满足以下长期目标：

1. **多人实时协同编辑**
   - 文档、Markdown、图表、任务树等内容支持多人同时编辑
   - 支持在线光标、选区、输入提示、当前编辑状态展示
   - 精确到字符区间的输入提示与在线状态同步

2. **团队与组织管理**
   - 支持用户注册登录
   - 支持多个组织（Organization）
   - 支持组织下多个 Team 分组
   - 支持类似 GitHub 的组织角色与资源权限管理

3. **资源统一管理**
   - Docs
   - Markdown
   - PDF（预览/批注/版本）
   - 图表 / 国策树 / 依赖图
   - 任务树
   - 文件资产库
   - 后续预留表格模块

4. **版本管理与操作记录**
   - 操作日志（细粒度）
   - 快照（Snapshot）
   - 提交记录（Commit）
   - 回滚
   - 历史对比

5. **多租户扩展**
   - 一个平台未来可服务多个组织
   - 从第一天开始就按多租户 SaaS 架构设计
   - 避免后期大规模返工

6. **国际化**
   - UI 文案不硬编码
   - 首期支持 `zh-CN` 与 `en-US`
   - 后续预留更多语言能力

7. **高品质 UI**
   - 苹果官网风格
   - 玻璃化
   - 动态渐变背景
   - 丰富的微交互
   - 明暗主题切换
   - 专业工作台式信息架构

## 3. 部署与运行约束

### 3.1 运行环境
- 操作系统：Linux
- 容器化：Docker
- 编排：Docker Compose
- 面板：宝塔 Linux
- 应用端口：`14010`
- 数据库端口：`3306`
- 配置管理：`.env`

### 3.2 更新方式
必须保证后续更新只需要执行：

```bash
docker compose -f docker-compose.yml up -d --build
```

因此系统设计必须满足：

- 所有服务通过 `docker-compose.yml` 编排
- 环境变量统一使用 `.env`
- 数据卷外置
- 数据迁移方案清晰
- 代码更新后可直接重建部署

### 3.3 数据库策略
- 主数据库固定为 **MySQL**
- 优先兼容宝塔可视化 MySQL 管理
- 除 MySQL 外尽可能自研，不依赖过重的外部中间件作为初期必要组件

## 4. 产品主轴：以 HOI4 Modding 为核心

相较于通用在线文档平台，本项目从第一天开始就明确偏向 **HOI4 Mod 制作协作平台**。

### 4.1 核心业务对象
平台未来需要围绕以下对象展开：

- HOI4 国策树（Focus Tree）
- HOI4 事件（Events）
- HOI4 决议（Decisions）
- HOI4 本地化文本（Localization）
- 设计文档（Docs / Markdown）
- 图标 / GFX / 素材资源
- 任务树 / 研发计划树
- 发布版本 / 提交记录 / 变更审阅

### 4.2 为什么要从一开始就围绕 HOI4 设计
如果一开始按纯通用文档平台建设，后期再加入 HOI4 结构化能力，会出现以下问题：

- 资源模型不适配
- 图编辑器语义不够
- 导出能力缺失
- 本地化数据结构难扩展
- 版本系统缺少领域上下文
- 权限与项目结构不能很好支撑 Mod 团队协作

因此架构必须从一开始就预埋：

- HOI4 资源类型
- HOI4 领域模块
- 图结构协同能力
- 导出 / 验证 / 打包接口
- 项目化结构管理

### 5.3 协同层

- 自研 Snapshot / Commit 版本体系

## 6. 为什么推荐 TypeScript 全栈

### 6.1 优势
1. 前后端统一语言，开发效率高
2. 非常适合 Web 实时协作场景
3. 前后端协议与类型共享更自然
4. 对多人协作产品、WebSocket、国际化、复杂 UI 更友好
5. Docker 化部署成熟
6. 后续 SaaS 化、多团队协作扩展成本较低

### 6.2 与 Java / Go 的比较结论
- **Java** 更偏企业后台，强在复杂业务系统，但对强实时 Web 协作产品不是首选
- **Go** 强在性能与并发，但不适合作为这一阶段全栈主力技术
- **TypeScript 全栈** 是当前项目的综合最优解

### 6.3 最终建议
- 前端：Next.js + React + TypeScript
- 后端：NestJS + TypeScript
- 数据库：MySQL
- 协同：自研实时同步层
- 部署：Docker Compose

## 7. 总体架构设计

### 7.1 第一阶段采用模块化单体
首期建议采用 **模块化单体 Monolith**，而不是直接微服务。

#### 原因
- 功能复杂但强关联
- 实时协作、权限、版本、资源之间耦合紧密
- 微服务会显著提高复杂度
- 更适合 Docker Compose 单机部署
- 有利于快速落地并保障一致性

### 7.2 后续可拆分服务
当访问量和协同规模扩大后，可逐步拆分：

- 协同同步服务
- 文件预处理服务
- 搜索服务
- 通知服务
- 导出/构建服务

- `hoi4_decision`

- `sheet`

不同资源类型不能共享一套完全相同的内部模型，因此需要插件式编辑器架构。

### 9.1 统一编辑器接口
每类资源编辑器实现统一能力接口：

- `loadResource`
- `saveDraft`
- `applyOperation`
- `getPresenceModel`
- `buildSnapshot`
- `exportResource`
- `validateResource`

### 9.2 编辑器类型
- 文档编辑器（Docs）
- Markdown 编辑器
- 图表编辑器
- HOI4 Focus Tree 编辑器
- HOI4 Localization 编辑器
- PDF 预览/批注编辑器

### 9.3 优势
- 后续新增编辑器不需要重构平台核心

- 更适合逐步扩展 HOI4 各类专用资源

## 11. 多租户与组织结构设计

### 11.1 层级模型
当前应采用如下结构：

- Platform
  - Organization
    - Repository
      - Project
        - Resource

这是本项目当前已经落地并验证通过的核心业务层级，整体逻辑参考 GitHub 的组织、成员、仓库思路，但会针对 HOI4 协作继续扩展项目与资源层。

### 11.2 关键能力
- 一个用户可加入多个 Organization
- Organization 内存在成员（Member）
- 成员可被分配一个或多个 Team Role
- Team Role 可作用于组织级、库级、项目级
- Project 必须隶属于 Repository
- Repository 必须隶属于 Organization
- Resource 必须隶属于 Organization、Repository 与 Project 的统一链路
- Studio 首页只展示当前用户所属的 Organization 内容

### 11.3 多租户原则
- 所有核心数据都带 `organization_id`
- 项目与资源进一步带 `repository_id`
- 资源访问必须先经过组织边界，再进入库与项目边界
- 系统从第一天就避免单租户写法
- 管理端、组织端、库端、项目端逻辑清晰分层

## 12. 用户、组织与权限模型

### 12.1 角色建议
平台当前采用两层角色模型：

#### 平台级
- Platform Admin
- User

#### 组织成员级
- Organization Owner
- Organization Admin
- Organization Maintainer
- Organization Editor
- Organization Viewer
- Guest

除此之外，还应引入 **Team Role** 作为可分配身份，用来承载更细粒度的项目权限，而不是把所有能力都塞进成员主角色。

### 12.2 权限模型
采用 **组织成员角色 + Team Role + Project Permission Assignment** 的组合模型。

#### 组织成员角色
处理组织层的基础管理能力，例如：
- 是否可以管理组织
- 是否可以邀请成员
- 是否可以创建库
- 是否可以创建项目

#### Team Role
处理成员在特定组织、库或项目下的身份，例如：
- 项目维护者
- 本地化编辑
- Focus Tree 设计维护
- 发布管理

#### 项目权限分配
处理更细粒度的能力，例如：
- `project_view`
- `project_edit`
- `project_delete`
- `resource_view`
- `resource_edit`
- `resource_delete`
- `focus_tree_view`
- `focus_tree_edit`
- `localization_view`
- `localization_edit`
- `collaboration_manage`
- `release_manage`

### 12.3 权限粒度
- 平台级
- 组织级
- 库级
- 项目级
- 资源级
- 操作级（查看 / 编辑 / 删除 / 导出 / 管理）

### 12.4 当前实现结论
当前数据模型已经明确支持：
- 用户加入多个组织
- 组织拥有成员列表
- 成员被分配 Team Role
- Team Role 决定项目能力范围
- Studio 只展示当前用户所属组织

## 17. 实时协作系统设计

这是整个平台技术上最核心的能力。

### 17.1 协作目标
- 支持多人同时编辑同一资源
- 在线成员实时可见
- 光标/选区/输入状态可见
- 修改低延迟同步
- 网络波动可恢复
- 历史操作可回放
- 不覆盖他人修改

### 17.2 协同引擎分层
#### TextCollabEngine
适用于：
- Docs
- Markdown
- Localization 值字段

#### GraphCollabEngine
适用于：
- Diagram
- HOI4 Focus Tree
- 任务树图结构部分

#### StructuredEntryCollabEngine
适用于：
- Localization Entry
- Event 字段编辑
- Decision 字段编辑

### 17.3 同步链路
1. 客户端进入资源房间
2. 获取最近快照
3. 拉取增量操作
4. 建立 WebSocket 连接
5. 本地乐观应用操作
6. 服务端校验并广播
7. 写入操作日志
8. 更新 Presence
9. 周期生成快照

### 17.4 Presence 模型
实时在线状态单独建模：
- 当前用户
- 当前所在资源
- 光标位置
- 选区范围
- 当前编辑块 / 节点 / 条目
- 最后活跃时间

### 17.5 冲突处理
- 文本：按操作位置映射与范围转换处理
- 图结构：按对象级操作处理
- 条目型：按条目级与字段级冲突处理

## 18. 版本系统设计

你明确希望类似 Git 的 commit 体验，因此版本层必须认真设计。

### 18.1 三层结构
#### Operation Log
最细粒度操作，例如：
- 插入字符
- 删除范围
- 修改节点位置
- 修改本地化条目值

#### Snapshot
周期性快照，用于快速恢复

#### Commit
由用户主动创建的人类可读提交，例如：
- `补充德意志国策树工业分支`
- `修复英国内阁事件链文案`
- `重构苏联决议依赖关系`

### 18.2 能力清单
- 操作历史回放
- 快照查看
- 提交说明
- 对比版本差异
- 回滚
- 从历史版本创建副本
- 审计谁修改了什么

## 19. 导出、验证与打包预留

HOI4 Modding 平台不能只停留在“在线编辑”，还必须能产出结构化成果。

### 19.1 必须预留的接口
- `exportResource`

## 22. 数据库设计思路（MySQL）

### 22.1 当前核心表建议
当前实际模型应围绕以下表展开：

#### 用户与认证
- `users`
- `user_emails`
- `auth_identities`
- `user_sessions`
- `verification_tokens`
- `password_reset_tokens`

#### 组织与成员
- `organizations`
- `organization_members`
- `organization_invites`
- `organization_sso_configs`
- `organization_domains`

#### 库、项目与权限
- `repositories`
- `team_roles`
- `team_role_permissions`
- `team_role_assignments`
- `projects`
- `project_permission_assignments`

#### 资源
- `resources`

这套表结构已经比最早的“组织 -> 项目 -> 资源”更进一步，升级为“组织 -> 库 -> 项目 -> 资源”，并把成员身份、邀请、认证、SSO 预留全部纳入。

### 22.2 通用字段原则
所有核心对象建议至少保留以下字段中的适用部分：
- `id`
- `organization_id`
- `repository_id`
- `project_id`
- `created_at`
- `updated_at`

资源类对象还应保留：
- `resource_type`
- `title`
- `slug`

权限与身份类对象还应保留：
- `role`
- `status`
- `scope`

### 22.3 设计原则
- 优先保证组织、库、项目三级边界清晰
- 身份与权限分层，不把所有权限混进单个角色字段
- Token 与会话独立建表
- 资源类型统一收口到 `resources`
- 索引从一开始就考虑组织、库、项目、资源类型维度
- 兼容后续快照、操作日志、评论、Presence、导出记录等扩展表

## 23. 登录注册与身份系统

### 23.1 当前身份结构
当前身份体系应按以下结构设计：

- `User`：账号主体
- `UserEmail`：邮箱列表与主邮箱关系
- `AuthIdentity`：第三方登录身份
- `UserSession`：登录会话
- `VerificationToken`：邮箱验证
- `PasswordResetToken`：重置密码
- `OrganizationInvite`：组织邀请

当前代码已经落地邮箱注册、密码登录、邮箱验证 token、忘记密码 token 预留，以及用户加入多个组织的基础结构。

### 23.2 首期支持
- 邮箱注册
- 密码登录
- 邮箱验证
- 忘记密码
- 组织邀请加入
- 用户加入多个组织

### 23.3 预留能力
- GitHub OAuth
- Google OAuth
- 企业 SSO（OIDC / SAML）

即使首期不全部开放，也必须从数据模型与服务接口层保留扩展能力。

### 23.4 当前实现说明
当前 Web 端仍以 Bearer Token 作为最小可用登录态方案，后续可继续演进为更完整的 Session / Refresh Token 策略，但不影响现阶段组织、成员与项目权限模型落地。

## 32. 最终结论

### 31.1 技术路线结论
本项目首选路线为：

- **前端：Next.js + React + TypeScript**
- **后端：NestJS + TypeScript**
- **数据库：MySQL**
- **协同：自研 WebSocket 同步层**
- **UI：苹果风格自研 Design System**
- **国际化：全量 i18n key 化**
- **部署：Docker Compose + .env + 宝塔 Nginx + 宝塔 MySQL**

平台不是泛化在线文档工具，而是：

> **以 HOI4 Modding 协作链路为核心的专业在线工作室平台。**

- 文档协作
- Focus Tree 设计
- Localization 管理
- 资源库
- 团队协作
- 提交与版本系统

### 31.3 架构结论
必须从第一天开始就做成：

- 多租户可扩展
- 资源类型可扩展
- 协同引擎可扩展
- 导出与验证能力可扩展
- HOI4 领域模型可扩展

这样才能保证后续增加事件、决议、角色、精神、包构建等功能时，不需要大规模推翻重构。
